# 로그 분석 리팩토링 유스케이스

## 문서 목적

이 문서는 리팩토링 전에 시스템이 만족해야 하는 실제 동작 시나리오를 정리한 문서다.

이후 구조 변경, 모듈 분리, parser 설계는 모두 이 문서를 기준으로 판단한다.

## 기본 사용자 시나리오

사용자의 기본 시나리오는 아래와 같다.

1. 사용자가 로그 파일을 업로드한다.
2. 시스템이 파일을 수신한다.
3. 시스템이 로그 포맷을 식별한다.
4. 시스템이 로그를 파싱한다.
5. 시스템이 통계 데이터를 계산한다.
6. 시스템이 분석 결과를 JSON 형태로 반환한다.

모든 유스케이스는 이 기본 흐름에서 어떤 단계가 확장되거나 달라지는지를 설명한다.

## 유스케이스 1. 기존에 없던 새 도메인과 새 로그 형식이 추가된다

이 유스케이스는 `Kokoa`, `Maver`의 변형이 아니라, 기존에 존재하지 않던 새 도메인과 새 로그 형식이 함께 추가되는 경우를 의미한다.

예:

- 새 외부 시스템이 추가된다.
- 그 시스템은 기존과 다른 로그 구조를 가진다.

### 기본 시나리오에서 달라지는 부분

3번과 4번 단계가 확장된다.

- 시스템은 새 도메인을 식별할 수 있어야 한다.
- 시스템은 새 도메인의 로그 형식을 파싱할 수 있어야 한다.

### 기대 결과

- 새 도메인 추가는 `parser` 모듈에 새 parser 구현체를 추가하는 방식으로 대응한다.
- core와 analysis는 최소 수정 또는 무수정 상태를 유지한다.

## 유스케이스 2. 기존 도메인에 새로운 로그 형식이 추가된다

이 유스케이스는 `Kokoa` 또는 `Maver` 같은 기존 도메인에 새로운 로그 형식이 추가되는 경우를 의미한다.

예:

- `Kokoa`가 기존 bracket 형식 외에 다른 형식을 추가한다.
- `Maver`가 기존 JSON 외에 다른 형식을 추가한다.

### 기본 시나리오에서 달라지는 부분

3번과 4번 단계가 확장된다.

- 시스템은 같은 도메인 안의 여러 형식을 구분해야 한다.
- 시스템은 도메인 parser 내부에서 해당 형식을 선택해 파싱해야 한다.

### 기대 결과

- parser 이름은 도메인 기준으로 유지한다.
- 새로운 형식은 기존 도메인 parser 내부 확장 또는 도메인 내부 형식 전략 추가 방식으로 대응한다.
- analysis는 로그 형식 변화와 무관하게 동작한다.

## 유스케이스 3. 시그니처로 도메인을 구분할 수 있다

로그 내용에 도메인을 식별할 수 있는 시그니처가 존재하는 경우다.

예:

- 특정 필드명
- 특정 URL 패턴
- 특정 prefix

### 기본 시나리오에서 달라지는 부분

3번 단계에서 시스템은 시그니처를 우선 검사한다.

### 기대 결과

- 시그니처만으로 도메인이 명확하면 파일명은 보지 않아도 된다.
- 시스템은 해당 도메인 parser를 선택하고 파싱을 진행한다.

## 유스케이스 4. 시그니처는 없지만 파일명으로 도메인을 구분할 수 있다

로그 내용만으로는 도메인을 구분할 수 없지만, 파일명은 도메인 판별에 도움이 되는 경우다.

예:

- 파일명이 `kokoa-20260331.log`다.
- 파일명이 `maver-access.log`다.

### 기본 시나리오에서 달라지는 부분

3번 단계에서 시스템은 시그니처 검사 후, 실패했을 때 파일명을 보조 힌트로 사용한다.

### 기대 결과

- 파일명은 2순위 힌트로만 사용한다.
- 파일명으로 도메인이 명확하면 해당 parser를 선택해 파싱한다.

## 유스케이스 5. 시그니처와 파일명이 모두 올바르며 서로 일치한다

로그 내용 시그니처와 파일명이 모두 존재하고, 둘이 같은 도메인을 가리키는 경우다.

### 기본 시나리오에서 달라지는 부분

3번 단계에서 시스템은 먼저 시그니처를 사용하고, 파일명은 추가 확인 정보로만 사용한다.

### 기대 결과

- 시그니처 기준으로 도메인을 확정한다.
- 파일명은 일치 여부 확인용 보조 정보로만 사용한다.
- 파싱은 정상 진행된다.

## 유스케이스 6. 파일명은 올바르지만 시그니처가 없다

파일명은 도메인 규칙에 맞지만, 로그 내용에는 도메인 시그니처가 없는 경우다.

### 기본 시나리오에서 달라지는 부분

3번 단계에서 시그니처 판별은 실패하고, 파일명 기반 판별로 넘어간다.

### 기대 결과

- 시스템은 파일명을 2순위 기준으로 사용한다.
- 파일명으로 도메인이 명확하면 파싱을 진행한다.

## 유스케이스 7. 시그니처는 있지만 파일명이 올바르지 않다

로그 내용에는 도메인 시그니처가 있지만, 파일명이 잘못되었거나 다른 도메인처럼 보이는 경우다.

### 기본 시나리오에서 달라지는 부분

3번 단계에서 시스템은 파일명보다 시그니처를 우선한다.

### 기대 결과

- 시스템은 시그니처 기준으로 도메인을 확정한다.
- 파일명 오류 때문에 파싱을 실패시키지 않는다.
- 필요하면 파일명 불일치 사실을 로그로 남긴다.

## 유스케이스 8. 시그니처와 파일명으로도 구분할 수 없다

로그 내용에 시그니처가 없고, 파일명도 도메인 판별에 도움이 되지 않는 경우다.

예:

- 파일명이 `access.log`처럼 일반적이다.
- 로그 구조가 여러 도메인에서 공통으로 사용할 수 있다.

### 기본 시나리오에서 달라지는 부분

3번 단계에서 시스템은 도메인을 식별하지 못한다.

### 기대 결과

- 시스템은 추정 파싱을 시도하지 않는다.
- 시스템은 명확한 실패를 반환한다.
- 실패 원인은 "시그니처와 파일명으로도 도메인을 식별할 수 없음"이어야 한다.

## 유스케이스 9. 파일명이 잘못되었고 시그니처도 없다

파일명도 잘못되었고 로그 내용에도 도메인 시그니처가 없는 최악의 경우다.

### 기본 시나리오에서 달라지는 부분

3번 단계에서 모든 도메인 판별이 실패한다.

### 기대 결과

- 시스템은 실패를 반환한다.
- parser는 억지로 선택되지 않는다.
- 사용자는 파일명 또는 로그 내용에 식별 가능한 정보가 필요하다는 사실을 알 수 있어야 한다.
